This is the group home page of the EPICS v4 Working Group.

EPICS v4 is a set of computer communications protocols, and a software framework, for high performance distributed control, message passing, and high level software services, as may be used in large scientific instruments and industrial plants. EPICS v4 was previously called "PvData", hence the name of the sourceforge project.

The EPICS v4 working group is a collaborative effort of members invited by Brookhaven Lab, to bring EPICS v4 to its full potential. BNL and PSI support is subject to the EPICS v4 Charter.

Code bases

Status 7-Nov-2011

The implementation statuses of the core packages of Epics v4, pvData, pvAccess, pvIOC, and PvService, given by language support, are:

Java

pvData, pvAccess, pvIOC, and PvService packages exist and are being developed as Eclipse projects, made available through sourceforge project epics-pvdata. The Java version of the pvIOC is now named "javaIOC". We have moved to a Mercurial repository, and Maven is doing all interna build and test

The following services have been started. Presently they can be found in PvService, though pvService is likely to be broken up into core support for services, and the services themselves removed:

  • Relational Database - code is complete and tested, about to enter deployment at PSI
  • itemFinder- Get a list of channel names and properties. Both client and server are implemented
  • gather service framework- get/put/monitor a set of V3 channels, Both client and server are implemented
  • modelling - a framework for hosting beamline optics modelling codes as services. The framework is complete, but no modelling is implemented.
  • Logging Service is being started

C++

pvDataCPP and pvAccessCPP are completed and available via sourceforge mercurial projects.

pvIOCCPP is started. It is implemented as EPICS V3 IOC applications. It has a pvAccess server that accesses EPICS V3 records. Much work remains.

pvServiceCPP has client side support for gather. It requires that a Java gather server be running.

Python

pvData and pvAccess - Guobao is working on wrapping pvDataCPP and pvAccessCPP

pvService - The client side of gather is implemented by using the C++ gather client support. Note about Python gather client:

A performance tests shows results equivalent to C++. This is because all the real work is done by C++ code not by Python. The V3 channel data is provided as a numPy 1-d double array that uses the underlying double array from the C++ array. It is expected that the Python client will pass the numPy array to other numPy or sciPy based Python code.

Meeting Minutes Standard

We record the minutes of the weekly telecon as a reference for people assigned action items, so they know the context of the required work, and for people outside the Working Group to see the bogus logic we used to reach conclusions.

  • Top of the minutes MUST include list of attendees, prefixed "PRESENT:", and the minute taker's id, prefixed "SCRIBE:", so that an automatic minute processor can remove the prefix of the minute taker from the published minutes. The chair will do a roll-call and do the PRESENT and SCRIBE entries
  • Minutes should try to include every (relevant) thought in the meeting
  • Action Items MUST be prefixed "AI" or "ACTION", and should include a person on which the action is assigned, and a due date. The form should be like "AI on Greg by 1/12: ..."
  • Specific decisions of the meeting MUST be prefixed "RESOLUTION:". The resolution must be fully described in the minutes, during the meeting. If the scribe isn't clear on the resolution, ask for clarification
  • Problematic areas of EPICS V4 should be prefixed "ISSUE:", and be described as completely as possible in the minutes, during the meeting.

AIs, RESOLUTIONS and ISSUES will be transcribed to sourceForge trackers by a chair following the meeting.
